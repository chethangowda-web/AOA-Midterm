Algorithm Basics
1.1 Notion of an Algorithm

An algorithm is a finite sequence of well-defined instructions to solve a problem in a finite amount of time. It takes input, processes it, and produces output.

Five key properties: Input, Output, Definiteness, Finiteness, Effectiveness.

Think of it as a recipe — step-by-step instructions with a clear start and end.

⚡ 30-Second Revision
  • Algorithm = Finite + Definite + Effective steps
  • Must terminate (finiteness is crucial!)
  • Each step must be unambiguous (definiteness)
  • Correctness must be provable
🎯 Frequently Asked by Examiners
  • Define algorithm and list its 5 properties (2-mark)
  • Difference between algorithm and program
  • Why finiteness is essential in an algorithm
1.2 Important Problem Types

Six major problem types exist in algorithms:

1. Sorting — Arrange elements in order
2. Searching — Find an element in a data structure
3. String Processing — Pattern matching in text
4. Graph Problems — Traversal, shortest path, spanning tree
5. Combinatorial Problems — Find combinations/permutations (TSP, Knapsack)
6. Geometric Problems — Convex hull, closest pair of points

⚡ 30-Second Revision
  • Mnemonic: S-S-S-G-C-G (Sorting, Searching, String, Graph, Combinatorial, Geometric)
  • Combinatorial problems are the hardest (exponential in worst case)
  • TSP and Knapsack are classic combinatorial problems
1.3 Order of Growth & Efficiency Classes

Order of growth describes how an algorithm's running time increases as input size n grows. We ignore constants and lower-order terms.

1.4 Analysis of Non-Recursive Algorithms

To analyze a non-recursive algorithm: identify the basic operation (the most frequently executed operation), set up a sum, and solve it.

Steps: (1) Decide input size measure, (2) Identify basic operation, (3) Check if cost varies for same n, (4) Set up sum, (5) Find closed form.

📐 Key Summation Formulas
Σ(i=1 to n) 1 = n
Σ(i=1 to n) i = n(n+1)/2
Σ(i=0 to n) 2ⁱ = 2ⁿ⁺¹ - 1
Σ(i=1 to n) i² = n(n+1)(2n+1)/6
Example: Selection Sort Analysis

1.5 Analysis of Recursive Algorithms

For recursive algorithms, set up a recurrence relation — an equation that expresses T(n) in terms of T of a smaller input. Then solve using backward substitution or Master's Theorem.

📐 Factorial Recurrence Example
T(n) = T(n-1) + 1, T(0) = 0
By backward substitution:
T(n) = T(n-1)+1 = T(n-2)+2 = ... = T(0)+n = n
∴ T(n) ∈ O(n)
Recursive Algorithm Analysis — Factorial
Factorial(n):
    if n == 0:
        return 1
    else:
        return n * Factorial(n-1)

// Recurrence: T(n) = T(n-1) + 1
// Solution: T(n) = n  → O(n)

Asymptotic Notations
2.1 Big-O Notation — O(g(n))

Big-O gives the upper bound (worst case). We say f(n) ∈ O(g(n)) if f(n) grows no faster than g(n) for large n.

Formally: f(n) ∈ O(g(n)) if ∃ positive constants c and n₀ such that f(n) ≤ c·g(n) for all n ≥ n₀.

Think: "f(n) is at most this bad."

📐 Big-O Definition
f(n) ∈ O(g(n)) iff ∃ c > 0, n₀ ≥ 0 such that
f(n) ≤ c · g(n) for all n ≥ n₀

Example: 5n² + 3n ∈ O(n²) [c=6, n₀=3]
⚡ 30-Second Revision — Big-O
  • O = Upper Bound = Worst Case
  • Drop constants and lower-order terms
  • 5n² + 3n + 7 → O(n²)
  • If f(n) = n, then f(n) ∈ O(n²) is also valid (not tight, but valid)
2.2 Omega Notation — Ω(g(n))

Omega gives the lower bound (best case). f(n) ∈ Ω(g(n)) means f(n) grows at least as fast as g(n).

Formally: f(n) ∈ Ω(g(n)) if ∃ positive constants c and n₀ such that f(n) ≥ c·g(n) for all n ≥ n₀.

📐 Omega Definition
f(n) ∈ Ω(g(n)) iff ∃ c > 0, n₀ ≥ 0 such that
f(n) ≥ c · g(n) for all n ≥ n₀

Example: 5n² + 3n ∈ Ω(n) [c=5, n₀=1]
2.3 Theta Notation — Θ(g(n))

Theta gives the tight bound (exact order). f(n) ∈ Θ(g(n)) means f(n) grows exactly as fast as g(n) — it's both O and Ω.

Formally: f(n) ∈ Θ(g(n)) if ∃ positive constants c₁, c₂ and n₀ such that c₁·g(n) ≤ f(n) ≤ c₂·g(n) for all n ≥ n₀.

📐 Theta Definition
f(n) ∈ Θ(g(n)) iff ∃ c₁, c₂ > 0, n₀ ≥ 0 such that
c₁·g(n) ≤ f(n) ≤ c₂·g(n) for all n ≥ n₀

Θ(g(n)) = O(g(n)) ∩ Ω(g(n))
Example: 5n² + 3n ∈ Θ(n²)

Brute Force Algorithms
3.1 Brute Force Technique

Brute Force is the most straightforward approach — try all possible solutions and pick the correct one. It's simple but usually inefficient.

Key idea: Apply the problem definition directly. No clever tricks, just raw computation.

Advantages: Simple to implement, always correct. Disadvantages: Very slow for large inputs.

⚡ 30-Second Revision
  • Brute Force = Direct application of problem definition
  • Always correct, rarely efficient
  • Useful as baseline for comparing better algorithms
  • Examples: Bubble Sort, Selection Sort, Linear Search, String Matching
3.2 Bubble Sort

Compare adjacent elements and swap if out of order. After each pass, the largest unsorted element "bubbles up" to its correct position.

After i passes, the last i elements are in their final position. Needs at most n-1 passes.

Pseudocode — Bubble Sort
BubbleSort(A[0..n-1]):
    for i = 0 to n-2:
        for j = 0 to n-2-i:
            if A[j] > A[j+1]:
                swap(A[j], A[j+1])

// Basic operation: comparison A[j] > A[j+1]
// Count = Σ(i=0 to n-2) Σ(j=0 to n-2-i) 1 = n(n-1)/2
// Time: Θ(n²) in all cases
CaseTimeSpace
BestO(n) with flag optimizationO(1)
AverageΘ(n²)O(1)
WorstΘ(n²)O(1)
🎯 Key Exam Points
  • Bubble Sort is stable (preserves relative order of equal elements)
  • Bubble Sort is in-place (O(1) auxiliary space)
  • With early termination flag: O(n) best case (already sorted)
  • Basic operation is the comparison, not the swap
3.3 Selection Sort

Find the minimum element in the unsorted portion and place it at the beginning. Repeat for remaining elements.

Unlike Bubble Sort, the number of swaps is minimized — at most n-1 swaps regardless of input. Makes it better for memory write-heavy scenarios.

Pseudocode — Selection Sort
SelectionSort(A[0..n-1]):
    for i = 0 to n-2:
        min_idx = i
        for j = i+1 to n-1:
            if A[j] < A[min_idx]:
                min_idx = j
        swap(A[i], A[min_idx])

// Basic operation: comparison A[j] < A[min_idx]
// Count = (n-1) + (n-2) + ... + 1 = n(n-1)/2
// Time: Θ(n²) always (input doesn't matter)
CaseTimeSwaps
All CasesΘ(n²)O(n)
⚠️ Key Difference: Selection Sort has exactly n(n-1)/2 comparisons always — the input doesn't affect this count. Bubble Sort can be O(n) in best case with flag. Selection Sort is NOT stable but Bubble Sort IS stable.
3.4 Linear Search

Scan each element one by one until the target is found or the list ends. The simplest search algorithm.

No requirement for sorted data — works on any list. Best for unsorted small arrays.

Pseudocode — Linear Search
LinearSearch(A[0..n-1], key):
    for i = 0 to n-1:
        if A[i] == key:
            return i
    return -1  // not found

// Best case: key at A[0] → O(1)
// Worst case: key at end or absent → O(n)
// Average case: O(n/2) ≈ O(n)
3.5 Brute Force String Matching

Given text T of length n and pattern P of length m, check every possible position in T where P could start, and verify character by character.

Try alignment at position 0, 1, 2, ..., n-m. For each, compare all m characters. Stop when match found or all positions tried.

Pseudocode — Brute Force String Matching
StringMatch(T[0..n-1], P[0..m-1]):
    for i = 0 to n-m:
        j = 0
        while j < m and T[i+j] == P[j]:
            j = j + 1
        if j == m:
            return i  // match at position i
    return -1

// Worst case: O(m × (n-m+1)) ≈ O(n·m)
// Example: T = "aaaa...a", P = "aaa...ab"
📐 Complexity Summary — Brute Force
Bubble Sort: Θ(n²) comparisons
Selection Sort: Θ(n²) comparisons, O(n) swaps
Linear Search: O(n) worst, O(1) best
String Match: O(n·m) worst case

Exhaustive Search
4.1 Exhaustive Search

Exhaustive Search (or generate-and-test) generates all candidate solutions and tests each for validity. Used for combinatorial problems where no better algorithm is known.

It is essentially brute force applied to combinatorial/optimization problems. Always finds the optimal solution, but at extreme cost for large inputs.

4.2 Travelling Salesperson Problem (TSP)

Given n cities and distances between them, find the shortest tour that visits each city exactly once and returns to the start city.

Brute force approach: Try all (n-1)! permutations of cities (fix one city), compute total distance, return the minimum. For 10 cities → 9! = 362,880 tours!

📐 TSP — Key Facts
Number of tours (for n cities) = (n-1)! / 2
(dividing by 2 removes reverse duplicates)

n = 4 cities → (4-1)!/2 = 3 tours
n = 10 cities → 9!/2 = 181,440 tours
Brute Force Complexity: O(n!)
TSP Example — 4 Cities
// Cities: 1, 2, 3, 4  (fix city 1 as start)
// Possible tours starting and ending at 1:
Tour 1: 1 → 2 → 3 → 4 → 1  cost = d(1,2)+d(2,3)+d(3,4)+d(4,1)
Tour 2: 1 → 2 → 4 → 3 → 1  cost = d(1,2)+d(2,4)+d(4,3)+d(3,1)
Tour 3: 1 → 3 → 2 → 4 → 1  cost = d(1,3)+d(3,2)+d(2,4)+d(4,1)

// Return tour with minimum cost
// Time: O(n!)  — impractical for n > 20

  • Why is TSP NP-Hard? (no polynomial algorithm known)
  • What is the time complexity of brute-force TSP?
  • How many distinct tours exist for n cities?
4.3 Knapsack Problem (Exhaustive Search)

Given n items each with a weight wᵢ and value vᵢ, and a knapsack of capacity W, select items to maximize total value without exceeding capacity W.

0/1 Knapsack: Each item is either taken (1) or not (0). Brute force: Enumerate all 2ⁿ subsets, compute total weight and value, keep feasible subset with max value.

📐 Knapsack — Key Facts
Number of subsets for n items = 2ⁿ
Brute Force Complexity: O(2ⁿ)

Feasibility: Total weight ≤ W
Objective: Maximize Σ vᵢ for selected items
Knapsack — Exhaustive Search Example
// n=3 items: (w1=2,v1=3), (w2=3,v2=4), (w3=4,v3=5), W=5
Subset {} → weight=0, value=0
Subset {1} → weight=2, value=3  ✓
Subset {2} → weight=3, value=4  ✓
Subset {3} → weight=4, value=5  ✓
Subset {1,2} → weight=5, value=7  ✓ ← MAX FEASIBLE
Subset {1,3} → weight=6, value=8  ✗ (exceeds W=5)
Subset {2,3} → weight=7, value=9  ✗
Subset {1,2,3} → weight=9, value=12  ✗

// Answer: Select items 1 and 2, value = 7
⚡ 30-Second Revision — TSP vs Knapsack
  • TSP: Find minimum cost Hamiltonian cycle → O(n!)
  • Knapsack: Maximize value within weight limit → O(2ⁿ) brute force
  • Both are NP-Hard optimization problems
  • Both require exhaustive enumeration in brute force approach
  • TSP: permutations | Knapsack: subsets

Divide and Conquer
5.1 General Method

Divide and Conquer solves a problem by: (1) Divide the problem into smaller subproblems, (2) Conquer by solving subproblems recursively, (3) Combine the solutions.

Subproblems are typically of the same type as the original. Works best when subproblems are independent and roughly equal in size.

📐 General Recurrence Equation
T(n) = aT(n/b) + f(n)

a = number of subproblems
b = factor by which problem size reduces
f(n) = cost of divide + combine steps
5.2 Binary Search

Works on sorted arrays only. Compare target with the middle element. If equal, return; if target is smaller, search left half; if larger, search right half. Eliminates half the elements each time.

Pseudocode — Binary Search (Iterative)
BinarySearch(A[0..n-1], key):
    low = 0,  high = n-1
    while low <= high:
        mid = (low + high) / 2
        if A[mid] == key: return mid
        elif A[mid] < key: low = mid + 1
        else: high = mid - 1
    return -1

// Recurrence: T(n) = T(n/2) + 1,  T(1) = 1
// Solution: T(n) = log₂(n) + 1 ∈ O(log n)

Master's Theorem
6.1 What is Master's Theorem?

Master's Theorem provides a direct formula to solve recurrences of the form T(n) = aT(n/b) + f(n) without manual substitution. It has three cases.

Think of it as comparing f(n) (the work done at each level) with n^(log_b a) (the work done by leaves of the recursion tree).

📐 Master's Theorem — 3 Cases
Given: T(n) = aT(n/b) + f(n), a ≥ 1, b > 1
Let p = log_b(a) [compare f(n) with nᵖ]

CASE 1: f(n) = O(nᵖ⁻ᵉ) for some ε > 0
→ T(n) = Θ(nᵖ) [leaves dominate]

CASE 2: f(n) = Θ(nᵖ)
→ T(n) = Θ(nᵖ log n) [equal work at each level]

CASE 3: f(n) = Ω(nᵖ⁺ᵉ) for some ε > 0
AND a·f(n/b) ≤ c·f(n) for c < 1 (regularity)
→ T(n) = Θ(f(n)) [root dominates]

⚡ 30-Second Revision — Master's Theorem Trick
  • Always compute p = log_b(a) first
  • Case 1: f(n) grows SLOWER than nᵖ → Answer is Θ(nᵖ)
  • Case 2: f(n) grows AT SAME RATE as nᵖ → Answer is Θ(nᵖ log n)
  • Case 3: f(n) grows FASTER than nᵖ → Answer is Θ(f(n))
  • Merge Sort: T(n)=2T(n/2)+n → p=1, f(n)=n → Case 2 → Θ(n log n)
🎯 Frequently Asked
  • State Master's Theorem with all three cases (4-mark)
  • Apply Master's Theorem to T(n) = 2T(n/2) + n

  • When does Master's Theorem NOT apply? (if f(n) is not polynomial)
⚠️ Master's Theorem does NOT apply when: f(n) is not polynomially comparable with n^(log_b a). Example: T(n) = 2T(n/2) + n/log n — this falls between Case 1 and 2 and cannot be solved directly with Master's Theorem.

Merge Sort & Quick Sort
7.1 Merge Sort

Divide: Split array into two equal halves. Conquer: Recursively sort each half. Combine: Merge the two sorted halves into one sorted array.

The merge step is the key — it takes two sorted arrays and produces one sorted array by comparing elements one by one.

Pseudocode — Merge Sort
MergeSort(A, left, right):
    if left < right:
        mid = (left + right) / 2
        MergeSort(A, left, mid)
        MergeSort(A, mid+1, right)
        Merge(A, left, mid, right)

Merge(A, left, mid, right):
    Create temp arrays L[] and R[]
    i=j=0, k=left
    while i < len(L) and j < len(R):
        if L[i] <= R[j]: A[k++] = L[i++]
        else: A[k++] = R[j++]
    Copy remaining elements

// Recurrence: T(n) = 2T(n/2) + n → Θ(n log n)

  • Always Θ(n log n) regardless of input
  • Stable sorting algorithm
  • Uses O(n) extra space — NOT in-place
  • Best for linked lists; preferred when stability is needed
  • Recurrence: T(n) = 2T(n/2) + n → Case 2 of Master's Theorem
7.2 Quick Sort

Divide: Choose a pivot element and partition array into two parts — elements less than pivot and elements greater than pivot. Conquer: Recursively sort both parts. Combine: Nothing — it's in-place!

The partition step is the key operation. Pivot ends up at its final sorted position after partitioning.

Pseudocode — Quick Sort
QuickSort(A, low, high):
    if low < high:
        pivot_idx = Partition(A, low, high)
        QuickSort(A, low, pivot_idx - 1)
        QuickSort(A, pivot_idx + 1, high)

Partition(A, low, high):  // Lomuto scheme
    pivot = A[high]
    i = low - 1
    for j = low to high-1:
        if A[j] <= pivot:
            i = i + 1
            swap(A[i], A[j])
    swap(A[i+1], A[high])
    return i + 1

⚠️ Quick Sort vs Merge Sort — Critical Differences:
Quick Sort: In-place (O(log n) stack), NOT stable, O(n²) worst case, faster in practice.
Merge Sort: O(n) extra space, Stable, Always O(n log n), better for linked lists.

Decrease and Conquer
8.1 General Method

Decrease and Conquer reduces a problem to a single smaller instance of the same problem (unlike Divide and Conquer which creates multiple subproblems).

Three types: Decrease by constant (n-1 each time), Decrease by factor (n/2 each time), Variable decrease (Euclid's GCD).

⚡ Key Difference: D&C vs Decrease&Conquer
  • Divide & Conquer: Split into a (a≥2) subproblems of size n/b
  • Decrease & Conquer: Reduce to ONE smaller problem (size n-1 or n/b)
  • D&C: Merge Sort → 2 subproblems
  • Dec&Conq: Insertion Sort → solve n-1, then insert nth element
8.2 Insertion Sort

Assume first i-1 elements are sorted. Take the i-th element and insert it into its correct position among the first i-1 elements. Repeat for i = 1 to n-1.

Like sorting playing cards in your hand — you pick one card at a time and slide it into the right place.

Pseudocode — Insertion Sort
InsertionSort(A[0..n-1]):
    for i = 1 to n-1:
        key = A[i]
        j = i - 1
        while j >= 0 and A[j] > key:
            A[j+1] = A[j]
            j = j - 1
        A[j+1] = key

// Best case: Already sorted → O(n) (inner while never executes)
// Worst case: Reverse sorted → O(n²)
// Basic op: comparison A[j] > key

  • Online algorithm — can sort as data arrives
  • Used in practice for small arrays (n ≤ 20)
  • Decrease-by-one: solve for n-1 elements, then insert nth
8.3 Topological Sorting

Given a Directed Acyclic Graph (DAG), topological sort produces a linear ordering of vertices such that for every edge (u→v), vertex u appears before v.

Used in scheduling tasks with dependencies. Example: course prerequisites, build systems, instruction scheduling.

Topological sort exists only for DAGs. If there's a cycle, no topological ordering is possible.

Algorithm — Source Removal Method (DFS-based)
TopologicalSort(G):
    // Method 1: Source removal
    result = []
    while G has vertices:
        Find a vertex v with in-degree = 0  // source
        Add v to result
        Remove v and all its edges from G
    return result

// Method 2: DFS-based
DFS_TopSort(G):
    Mark all vertices as unvisited
    result = empty stack
    for each unvisited vertex v:
        DFS(v, result)
    Pop all from stack // gives topological order

// Time: O(V + E) — same as DFS
📐 Topological Sort — Key Facts
Works only on: DAG (Directed Acyclic Graph)
Time Complexity: O(V + E)
Space: O(V)

If in-degree of a vertex = 0 → it's a source → it can come first
If cycle exists → no topological ordering → algorithm fails

// Graph: A→C, A→B, B→D, C→D, D→E

        Update: D=1

        Update: D=0

Q1 Define Big-O notation formally with example.

Definition: f(n) ∈ O(g(n)) iff ∃ positive constants c and n₀ such that f(n) ≤ c·g(n) for all n ≥ n₀.

  • Gives upper bound (worst-case behavior)
  • Example: 3n + 5 ∈ O(n) — choose c = 4, n₀ = 5. Then 3n+5 ≤ 4n ∀n≥5 ✓
  • O(n²) is also valid for 3n+5, just not tight
Q2 Define Omega (Ω) notation with example.

Definition: f(n) ∈ Ω(g(n)) iff ∃ positive constants c and n₀ such that f(n) ≥ c·g(n) for all n ≥ n₀.

  • Gives lower bound — algorithm takes at least this much time
  • Example: 5n² + 3n ∈ Ω(n²) — choose c = 5, n₀ = 1. Then 5n²+3n ≥ 5n² ∀n≥1 ✓
  • Ω(1) holds for every algorithm
Q3 Define Theta (Θ) notation with example.

Definition: f(n) ∈ Θ(g(n)) iff ∃ positive constants c₁, c₂, n₀ such that c₁·g(n) ≤ f(n) ≤ c₂·g(n) for all n ≥ n₀.

  • Gives tight bound — exact order of growth
  • Θ(g(n)) = O(g(n)) ∩ Ω(g(n))
  • Example: 5n² + 3n ∈ Θ(n²) — c₁=5, c₂=6, n₀=3
  • Most precise notation — preferred in analysis
Q4 What is the difference between O and Θ? Why does it matter?
  • O = upper bound: f could be faster (not tight)
  • Θ = tight bound: f grows exactly at this rate
  • Example: Bubble Sort is both O(n²) AND O(n³) and O(2ⁿ) — all valid but not tight
  • Bubble Sort is Θ(n²) — this is the precise statement
  • Examiners expect Θ when asking for "exact" complexity
Q5 Prove: 5n² + 3n ∈ O(n²).

We need to find c > 0, n₀ such that 5n² + 3n ≤ c·n² for all n ≥ n₀.

  • Choose c = 6: 5n² + 3n ≤ 6n² iff 3n ≤ n² iff 3 ≤ n
  • So for c = 6, n₀ = 3: 5n² + 3n ≤ 6n² ∀n ≥ 3 ✓
  • Therefore 5n² + 3n ∈ O(n²)
Q6 Simplify: T(n) = 7n³ + 4n² + 2n + 9. Give tightest bound.
  • Dominant term = n³ (grows fastest)
  • Drop all lower-order terms and constants
  • T(n) ∈ Θ(n³)
  • Also valid: T(n) ∈ O(n³), T(n) ∈ Ω(n³)
  • NOT correct to say Θ(n²) or Θ(n⁴) — those are not tight
Q7 State the transitivity and sum rules of asymptotic notation.

Transitivity: If f(n) ∈ O(g(n)) and g(n) ∈ O(h(n)), then f(n) ∈ O(h(n)).

Sum Rule: If f₁(n) ∈ O(g₁(n)) and f₂(n) ∈ O(g₂(n)), then f₁(n) + f₂(n) ∈ O(max(g₁(n), g₂(n))).

  • Example: O(n²) + O(n³) = O(n³)
  • Example: O(n log n) + O(n²) = O(n²)
Q8 What is the relationship: n! vs 2ⁿ vs n² — which grows fastest?
  • Growth order: n! > 2ⁿ > n³ > n² > n log n > n > log n > 1
  • n! grows fastest (factorial). For n = 20: 20! ≈ 2.4 × 10¹⁸
  • 2ⁿ grows next. For n = 20: 2²⁰ = 1,048,576
  • This is why TSP (O(n!)) is far worse than Knapsack brute force (O(2ⁿ))
Q9 Is O(log₂n) same as O(log₁₀n)? Explain.
  • Yes, they are asymptotically the same: O(log₂n) = O(log₁₀n)
  • Because: log₂n = log₁₀n / log₁₀2 = log₁₀n × 3.32...
  • The base change factor (3.32) is a constant — absorbed by Big-O
  • In Big-O, we simply write O(log n) without specifying base
Q10 What is the tightest complexity class for Binary Search?
  • Binary Search: T(n) = T(n/2) + 1, T(1) = 1
  • Solved: T(n) = log₂n + 1
  • Tightest bound: Θ(log n)
  • Best case: O(1) (key is middle element)
  • Worst/Average case: Θ(log n)
  • Space: O(1) iterative, O(log n) recursive (call stack)
Q1 Define brute force technique. Give two examples.

Brute Force = Directly apply the problem definition, trying all candidates without clever tricks.

  • Simple, straightforward, always correct
  • Example 1: Bubble Sort — compare all adjacent pairs and swap
  • Example 2: String Matching — check pattern at every text position
  • Weakness: Often O(n²) or worse — inefficient for large inputs
Q2 What is the time complexity of Bubble Sort and how is it derived?

Bubble Sort: Θ(n²)

  • Outer loop: i = 0 to n–2 (runs n–1 times)
  • Inner loop: j = 0 to n–2–i (runs n–1–i times)
  • Total comparisons = Σ(n–1–i) from i=0 to n–2 = (n–1)+(n–2)+...+1 = n(n–1)/2
  • n(n–1)/2 ∈ Θ(n²) ✓
Q3 Why is Selection Sort always Θ(n²) regardless of input?
  • Selection Sort must always find the minimum in the unsorted portion
  • This requires scanning ALL remaining elements — no shortcut possible
  • Total comparisons = (n–1)+(n–2)+...+1 = n(n–1)/2 — always the same count
  • Unlike Bubble Sort (which can exit early with flag), Selection Sort has no early exit
  • ∴ Best = Worst = Average = Θ(n²)
Q4 What is the worst case of brute force string matching and when does it occur?
  • Worst case: O(n × m) where n = text length, m = pattern length
  • Occurs when: Almost every position in text nearly-matches the pattern
  • Classic example: Text = "aaaaaa...a", Pattern = "aaa...ab"
  • At every position, m–1 characters match before failure → m comparisons per shift
  • n–m+1 positions × m comparisons = O(n·m)
Q5 Is Bubble Sort stable? Is Selection Sort stable? Justify.
  • Bubble Sort — Stable ✓: Only swaps when A[j] > A[j+1] (strictly greater). Equal elements never swap → relative order preserved
  • Selection Sort — NOT stable ✗: Swapping the minimum with A[i] can skip over equal elements
  • Example: [3a, 3b, 1] → Selection sort → [1, 3b, 3a]: 3a and 3b swapped — unstable!
Q6 What is the number of swaps in Selection Sort? Why is this an advantage?
  • Selection Sort does at most n–1 swaps (one per outer loop iteration)
  • This is O(n) — optimal for swaps
  • Advantage: When memory writes are expensive (e.g., flash memory), fewer writes = better
  • Compared to Bubble Sort which can do O(n²) swaps in worst case
  • Trade-off: Same O(n²) comparisons, but far fewer writes
Q7 What is Linear Search? Give its best and worst case complexities.
  • Linear Search = Scan each element one by one until key is found or list ends
  • Works on unsorted arrays (no precondition needed)
  • Best case: O(1) — key is at first position
  • Worst case: O(n) — key is last or absent
  • Average case: O(n/2) ≈ O(n)
  • Space: O(1)
Q8 Differentiate Bubble Sort and Selection Sort.
  • Bubble Sort: Compares adjacent pairs, swaps immediately, stable, O(n²) swaps worst case, can detect sorted input (O(n) best with flag)
  • Selection Sort: Finds minimum, single swap per pass, NOT stable, only O(n) swaps, always Θ(n²)
  • Bubble Sort is adaptive; Selection Sort is not
Q9 What is the space complexity of Bubble Sort and Selection Sort?
  • Both are O(1) space — in-place sorting algorithms
  • No extra arrays needed; only a few variables (loop counters, temp for swap)
  • In-place = does not require extra memory proportional to input size
  • Compare: Merge Sort uses O(n) extra space — NOT in-place
Q10 Give pseudocode outline for brute force string matching and its complexity.

Pseudocode: for i = 0 to n–m: if T[i..i+m–1] == P[0..m–1]: return i. return –1

  • Outer loop: n–m+1 positions
  • Inner comparison: up to m characters each
  • Best case: O(m) — match at first position
  • Worst case: O((n–m+1)×m) ≈ O(n·m)
  • Average case: O(n) for random text
Q1 Define Travelling Salesperson Problem (TSP).
  • Given: n cities with pairwise distances (complete weighted graph)
  • Find: Shortest Hamiltonian cycle — visit each city exactly once and return to start
  • Decision version: Is there a tour of cost ≤ k?
  • Optimization version: Find the minimum cost tour
  • TSP is NP-Hard — no known polynomial-time algorithm
Q2 How many distinct tours in TSP for n cities (brute force)?
  • Fix one city (say city 1) to avoid counting rotations of same tour
  • Remaining (n–1) cities can be arranged in (n–1)! ways
  • Each tour counted twice (forward + reverse) → (n–1)!/2 distinct tours
  • n=4: (4–1)!/2 = 3. n=5: 4!/2 = 12. n=10: 9!/2 = 181,440
  • Complexity: O(n!)
Q3 Define the 0/1 Knapsack Problem.
  • Given: n items, each with weight w[i] and value v[i]; knapsack capacity W
  • Goal: Select items to maximize total value such that total weight ≤ W
  • Each item: either fully taken (1) or left (0) — no fractions
  • Brute force: enumerate all 2ⁿ subsets
  • Applications: cargo loading, resource allocation, investment planning
Q4 What is the brute force complexity of 0/1 Knapsack?
  • n items → 2ⁿ possible subsets (each item: take or leave)
  • For each subset: check feasibility (weight ≤ W) and record max value
  • Complexity: O(2ⁿ) — exponential
  • For n=20: 2²⁰ ≈ 1 million. For n=50: 2⁵⁰ ≈ 10¹⁵ — impractical
  • Better solution: Dynamic Programming reduces this to O(nW)
Q5 What is exhaustive search? When is it used?
  • Exhaustive search = Generate all candidate solutions, test each, keep the best
  • Also called generate and test
  • Used when: problem has small input, no better algorithm known, need exact optimal solution
  • Examples: TSP (all permutations), Knapsack (all subsets), Subset Sum
  • Advantage: Guaranteed optimal. Disadvantage: Exponential time
Q6 What is the difference between TSP and Knapsack in brute force?
  • TSP: Combinatorial on permutations — enumerate all (n–1)! tours → O(n!)
  • Knapsack: Combinatorial on subsets — enumerate all 2ⁿ subsets → O(2ⁿ)
  • TSP is harder: n! >> 2ⁿ for large n
  • TSP: minimize tour cost. Knapsack: maximize value within weight
  • Both: NP-Hard problems
Q7 What is a Hamiltonian cycle?
  • A Hamiltonian cycle is a cycle in a graph that visits every vertex exactly once and returns to the starting vertex
  • Not all graphs have Hamiltonian cycles
  • TSP seeks the minimum cost Hamiltonian cycle in a complete weighted graph
  • Hamiltonian path: visits every vertex once but doesn't return to start
  • Finding Hamiltonian cycle is NP-Complete
Q8 Solve Knapsack for: Items = {(w=2,v=3),(w=3,v=4),(w=4,v=5)}, W=5.

Enumerate all 2³ = 8 subsets:

  • {} → w=0, v=0 ✓
  • {1} → w=2, v=3 ✓
  • {2} → w=3, v=4 ✓
  • {3} → w=4, v=5 ✓
  • {1,2} → w=5, v=7 ✓ ← MAX
  • {1,3} → w=6 ✗ (exceeds W)
  • {2,3} → w=7 ✗
  • {1,2,3} → w=9 ✗

Answer: Select items 1 and 2, value = 7

Q9 Why is TSP called NP-Hard?
  • No known polynomial-time algorithm can solve TSP optimally
  • Every known exact algorithm has exponential worst-case complexity
  • NP-Hard = at least as hard as the hardest problems in NP
  • Even verifying a solution is "optimal" requires checking all others
  • Approximation algorithms exist (e.g., Christofides: 1.5× optimal in polynomial time)
Q10 What is the role of exhaustive search in algorithm design?
  • Serves as a baseline for comparing efficient algorithms
  • Used to verify correctness of faster algorithms on small inputs
  • Foundation for understanding problem difficulty
  • Leads to development of better strategies: greedy, DP, branch-and-bound
  • Always correct — useful when n is very small (n ≤ 20 for 2ⁿ, n ≤ 12 for n!)
Q1 State the general recurrence for Divide and Conquer.
  • T(n) = aT(n/b) + f(n)
  • a = number of subproblems (a ≥ 1)
  • b = factor by which problem size reduces (b > 1)
  • f(n) = cost of divide + combine steps
  • Example: Merge Sort → a=2, b=2, f(n)=n → T(n)=2T(n/2)+n
Q2 What is Binary Search? State its recurrence and complexity.
  • Binary Search: Compare target with middle element of sorted array; recurse on left or right half
  • Recurrence: T(n) = T(n/2) + 1, T(1) = 1
  • Solution: T(n) = log₂n + 1 ∈ Θ(log n)
  • Best: O(1), Worst: O(log n), Space: O(1)
  • Requires sorted input — limitation!
Q3 What is the recurrence for Merge Sort? Solve it.
  • Merge Sort: T(n) = 2T(n/2) + n, T(1) = 0
  • By Master's Theorem: a=2, b=2, p = log₂2 = 1, nᵖ = n. f(n) = n = Θ(n) → Case 2
  • ∴ T(n) = Θ(n log n)
  • This is the same for best, average, and worst case
Q4 When does Quick Sort degrade to O(n²)?
  • When pivot is always the smallest or largest element → most unbalanced partition
  • Happens with: already sorted array (ascending or descending) when first/last element is pivot
  • Recurrence (worst case): T(n) = T(n–1) + T(0) + n = T(n–1) + n → O(n²)
  • Solution: Use random pivot or median-of-three pivot selection
Q5 Compare Merge Sort and Quick Sort briefly.
  • Merge Sort: Always Θ(n log n), O(n) extra space, Stable, Good for linked lists
  • Quick Sort: O(n log n) avg, O(n²) worst, O(log n) space, NOT stable, faster in practice
  • Merge Sort = predictable performance. Quick Sort = better cache performance on arrays
Q6 State Master's Theorem Case 2 and give an example.
  • Case 2: If f(n) = Θ(n^(log_b a)), then T(n) = Θ(n^(log_b a) × log n)
  • i.e., when the cost at each level is equal — log n levels → multiply by log n
  • Example: T(n) = 2T(n/2) + n → a=2, b=2, log₂2 = 1, f(n)=n=Θ(n¹) → Case 2 → Θ(n log n)
  • Also: T(n) = T(n/2) + 1 → log₂1=0, f(n)=1=Θ(n⁰) → Case 2 → Θ(log n)
Q7 What is a stable sorting algorithm? Which of Merge, Quick, Bubble are stable?
  • Stable sort: Preserves the relative order of equal elements
  • Example: [3a, 1, 3b] sorted → [1, 3a, 3b] (stable) not [1, 3b, 3a]
  • Bubble Sort — Stable ✓
  • Merge Sort — Stable ✓ (merge step: take left element first on tie)
  • Quick Sort — NOT Stable ✗ (partition step can reorder equals)
Q8 Apply Master's Theorem to T(n) = 4T(n/2) + n².
  • a=4, b=2, log_b(a) = log₂(4) = 2, so nᵖ = n²
  • f(n) = n² = Θ(n²) = Θ(nᵖ) → Case 2
  • ∴ T(n) = Θ(n² log n)
Q9 What is the space complexity of Quick Sort?
  • Quick Sort is in-place — does NOT use extra array
  • But it uses O(log n) space for the recursive call stack (average case)
  • Worst case stack depth: O(n) — when always unbalanced (already sorted + last pivot)
  • Compare: Merge Sort uses O(n) extra space for temp arrays + O(log n) stack
Q10 What are the three steps of Divide and Conquer?
  • 1. Divide – Break problem into a smaller subproblems of size n/b
  • 2. Conquer – Recursively solve each subproblem (base case: solve directly)
  • 3. Combine – Merge/combine solutions to subproblems to get overall solution
  • Example: Merge Sort — Divide array, sort halves, Merge sorted halves
  • Binary Search has no combine step (only one subproblem to recurse on)
Q1 Differentiate Divide and Conquer from Decrease and Conquer.
  • Divide & Conquer: Splits into a ≥ 2 subproblems; solutions combined. Example: Merge Sort
  • Decrease & Conquer: Reduces to ONE smaller subproblem; no combine needed. Example: Insertion Sort
  • D&C recurrence: T(n) = aT(n/b)+f(n). Dec&Conq: T(n) = T(n–1) + f(n)
Q2 Why is Insertion Sort a Decrease-and-Conquer algorithm?
  • To sort n elements: first solve for n–1 elements (same smaller problem), then insert nth
  • Problem size decreases by 1 each step → "decrease by constant"
  • No combine step needed — just insert into correct position
  • Recurrence: T(n) = T(n–1) + n → Θ(n²)
Q3 What is topological sorting? On what type of graph?
  • Linear ordering of vertices of a DAG (Directed Acyclic Graph) such that for every edge u→v, u appears before v
  • Exists only for DAGs — if cycle present, no topological order
  • Multiple valid orderings may exist
  • Applications: task scheduling, course prerequisites, build systems
Q4 What is the time complexity of Insertion Sort? Best and worst case?
  • Best case: O(n) — Input already sorted; inner while loop never executes
  • Worst case: O(n²) — Input reverse sorted; maximum shifting at each step
  • Average case: O(n²)
  • Basic operation: comparison A[j] > key
  • Space: O(1) — in-place, stable
Q5 What is the time complexity of topological sort?
  • O(V + E) — same as DFS/BFS
  • V = vertices visited once each → O(V)
  • E = each edge processed once → O(E)
  • Source removal method: computing in-degrees O(V+E) + V removals = O(V+E)
  • DFS method: DFS itself is O(V+E)
Q6 What are the three types of Decrease and Conquer?
  • Decrease by constant (by 1): Problem size reduces by 1 each step. Example: Insertion Sort (n → n–1), Factorial recursion
  • Decrease by constant factor: Size reduces by factor b. Example: Binary Search (n → n/2)
  • Variable decrease: Reduction amount varies. Example: Euclid's GCD algorithm
Q7 Trace one pass of Insertion Sort on [8, 4, 23, 42, 16].
  • i=1, key=4: Compare 4 < 8 → shift 8 right → [4, 8, 23, 42, 16]
  • i=2, key=23: 23 > 8, stop → [4, 8, 23, 42, 16]
  • i=3, key=42: 42 > 23, stop → [4, 8, 23, 42, 16]
  • i=4, key=16: 16 < 42 → shift. 16 < 23 → shift. 16 > 8, stop → [4, 8, 16, 23, 42]
  • Final sorted: [4, 8, 16, 23, 42]
Q8 What is an in-degree in a graph? Why is it used in topological sort?
  • In-degree of vertex v = number of incoming edges to v
  • A vertex with in-degree 0 has no prerequisites → can be placed first in topological order
  • Source removal algorithm: repeatedly remove vertices with in-degree 0
  • After removing a vertex, update in-degrees of its neighbors
  • If any vertex still has in-degree > 0 at end → cycle exists → no topological sort
Q9 What happens when topological sort is applied to a graph with a cycle?

Q1 Explain in detail the asymptotic notations Big-O, Omega, and Theta. State formal definitions, properties, and prove that 5n² + 3n ∈ Θ(n²). [10 marks]
1. Introduction (1 mark)

Asymptotic notations describe the growth rate of an algorithm's time/space complexity as input n → ∞. They classify algorithms by how their running time scales, ignoring constants and lower-order terms. The three main notations are Big-O, Omega, and Theta.

2. Big-O Notation — Upper Bound (2 marks)

Definition: f(n) ∈ O(g(n)) iff ∃ positive constants c and n₀ such that f(n) ≤ c·g(n) for all n ≥ n₀.

  • Gives the worst-case upper bound — algorithm takes at most this long
  • Example: 3n + 5 ∈ O(n). Choose c = 4, n₀ = 5. Then 3n+5 ≤ 4n ∀n ≥ 5 ✓
  • Not unique: 3n+5 ∈ O(n²) is also valid — just not tight
3. Omega Notation — Lower Bound (2 marks)

Definition: f(n) ∈ Ω(g(n)) iff ∃ positive constants c and n₀ such that f(n) ≥ c·g(n) for all n ≥ n₀.

  • Gives the best-case lower bound — algorithm takes at least this long
  • Example: 5n² + 3n ∈ Ω(n²). Choose c = 5, n₀ = 1. Then 5n²+3n ≥ 5n² ∀n ≥ 1 ✓
4. Theta Notation — Tight Bound (2 marks)

Definition: f(n) ∈ Θ(g(n)) iff ∃ positive constants c₁, c₂, n₀ such that c₁·g(n) ≤ f(n) ≤ c₂·g(n) for all n ≥ n₀.

  • Θ(g(n)) = O(g(n)) ∩ Ω(g(n)) — both upper and lower bound simultaneously
  • Most precise: f grows exactly at the rate of g
5. Proof: 5n² + 3n ∈ Θ(n²) (2 marks)

Need to find c₁, c₂, n₀ such that c₁·n² ≤ 5n² + 3n ≤ c₂·n²

  • Upper bound: 5n² + 3n ≤ c₂·n² → 5 + 3/n ≤ c₂. For n ≥ 1: 5 + 3/n ≤ 8. Choose c₂ = 8, n₀ = 1 ✓
  • Lower bound: c₁·n² ≤ 5n² + 3n. Choose c₁ = 5. Then 5n² ≤ 5n² + 3n ∀n ≥ 0 ✓
  • For c₁ = 5, c₂ = 8, n₀ = 1: 5n² ≤ 5n² + 3n ≤ 8n² ∀n ≥ 1
  • ∴ 5n² + 3n ∈ Θ(n²) ✓
6. Summary Table (1 mark)

O: f ≤ c·g (upper). Ω: f ≥ c·g (lower). Θ: c₁·g ≤ f ≤ c₂·g (tight). Θ is used when we know exact growth rate. O used for worst case guarantees.

Q3 What is brute force string matching? Explain with algorithm and example. What are its limitations? Compare with better approaches. [10 marks]
1. Definition (1 mark)

Brute force string matching: Given text T[0..n-1] and pattern P[0..m-1], check every possible starting position i in T (from 0 to n–m) to see if P occurs there. Also called the "naive" string matching algorithm.

2. Algorithm / Pseudocode (2 marks)

BruteForceStringMatch(T[0..n-1], P[0..m-1]):
  for i = 0 to n–m:
    j = 0
    while j < m and T[i+j] == P[j]: j++
    if j == m: return i // match found at i
  return –1 // no match

  • Outer loop tries each of n–m+1 starting positions
  • Inner loop compares up to m characters
3. Traced Example (2 marks)

T = "ABCABDABD", P = "ABD" (n=9, m=3)

  • i=0: A=A, B=B, C≠D → fail
  • i=1: B≠A → fail
  • i=2: C≠A → fail
  • i=3: A=A, B=B, D=D → MATCH at position 3!
4. Complexity Analysis (2 marks)
  • Best case: O(m) — Match at first position, or mismatch on 1st char always
  • Worst case: O((n–m+1)·m) ≈ O(n·m)
  • Worst case example: T = "aaaaa...a" (n a's), P = "aaa...ab" (m–1 a's + 1 b)
  • At every position: m–1 matches then failure → O(n·m) total
  • Average case (random text): O(n)
5. Limitations (1 mark)
  • No learning from previous comparisons — no memory of past matches
  • Redundant comparisons — shifts back on every mismatch
  • Inefficient for large texts or large patterns
6. Better Algorithms (2 marks)
  • KMP (Knuth-Morris-Pratt): O(n+m) — uses failure function to avoid redundant comparisons
  • Boyer-Moore: O(n/m) best case — scans right to left, skips large chunks
  • Rabin-Karp: O(n+m) average — uses hashing for fast match checking
  • Brute force is still used for short patterns or small n due to simplicity

2. Exhaustive Search Approach (2 marks)

  • Enumerate all (n–1)! permutations of remaining cities
  • Compute cost of each tour
  • Return the minimum cost tour

Q6 State and explain Master's Theorem with all three cases. Apply it to solve: (i) T(n)=2T(n/2)+n, (ii) T(n)=4T(n/2)+n², (iii) T(n)=4T(n/2)+n³. [10 marks]
1. Master's Theorem — Statement (2 marks)

Given recurrence: T(n) = aT(n/b) + f(n) where a ≥ 1, b > 1.

Let p = log_b(a). Compare f(n) with nᵖ:

  • Case 1: f(n) = O(nᵖ⁻ᵉ) for some ε > 0 → T(n) = Θ(nᵖ) [leaves dominate]
  • Case 2: f(n) = Θ(nᵖ) → T(n) = Θ(nᵖ log n) [all levels equal]
  • Case 3: f(n) = Ω(nᵖ⁺ᵉ) AND a·f(n/b) ≤ c·f(n), c<1 → T(n) = Θ(f(n)) [root dominates]
2. Application (i): T(n) = 2T(n/2) + n (2 marks)
  • a = 2, b = 2, p = log₂(2) = 1, so nᵖ = n
  • f(n) = n. Compare: f(n) = n = Θ(n) = Θ(nᵖ) → Case 2
  • ∴ T(n) = Θ(n log n)
  • This is Merge Sort! ✓
3. Application (ii): T(n) = 4T(n/2) + n² (2 marks)
  • a = 4, b = 2, p = log₂(4) = 2, so nᵖ = n²
  • f(n) = n². Compare: f(n) = n² = Θ(n²) = Θ(nᵖ) → Case 2
  • ∴ T(n) = Θ(n² log n)
4. Application (iii): T(n) = 4T(n/2) + n³ (2 marks)
  • a = 4, b = 2, p = log₂(4) = 2, so nᵖ = n²
  • f(n) = n³. Compare: n³ vs n² → n³ grows faster → f(n) = Ω(n²⁺¹) → ε = 1 → Case 3
  • Regularity check: 4·f(n/2) = 4·(n/2)³ = n³/2 = (1/2)·n³ ≤ c·n³, c = 1/2 < 1 ✓
  • ∴ T(n) = Θ(n³)

1. Concept — Divide and Conquer Approach (1 mark)

Merge Sort uses the divide and conquer paradigm: Divide array into two halves → Conquer by recursively sorting each half → Combine by merging two sorted halves into one sorted array.

2. Pseudocode (3 marks)

MergeSort(A, left, right):
  if left < right:
    mid = (left + right) / 2
    MergeSort(A, left, mid)
    MergeSort(A, mid+1, right)
    Merge(A, left, mid, right)

Merge(A, left, mid, right):
  Create temp arrays L = A[left..mid], R = A[mid+1..right]
  i = j = 0; k = left
  while i < len(L) and j < len(R):
    if L[i] ≤ R[j]: A[k++] = L[i++] else: A[k++] = R[j++]
  Copy remaining of L and R into A

  • [43,3] → [43] | [3] → Merge → [3,43]
  • Merge [27,38] + [3,43] → [3,27,38,43]
  • Right: [9,82,10] → [9] | [82,10]
  • [82,10] → Merge → [10,82]
  • Merge [9] + [10,82] → [9,10,82]
  • Final: Merge [3,27,38,43] + [9,10,82] → [3,9,10,27,38,43,82] ✓

1. Concept (1 mark)

Quick Sort: Choose a pivot element. Partition array so all elements less than pivot are on left, greater on right. Pivot is then in its final sorted position. Recursively sort left and right parts. No combine step needed — in-place!

2. Pseudocode (3 marks)

QuickSort(A, low, high):
  if low < high:
    pivot_pos = Partition(A, low, high)
    QuickSort(A, low, pivot_pos – 1)
    QuickSort(A, pivot_pos + 1, high)

Partition(A, low, high): [Lomuto scheme — pivot = A[high]
  pivot = A[high]; i = low – 1
  for j = low to high – 1:

  return i + 1

Q9 Explain Decrease and Conquer technique. Describe Insertion Sort and Topological Sorting with pseudocode, examples, and complexity analysis. [10 marks]
1. Decrease and Conquer — General Method (2 marks)

Decrease and Conquer reduces a problem of size n to a single problem of size n–1 (or n/b). Three variants:

  • Decrease by constant: Size reduces by fixed amount (usually 1). Example: Insertion Sort
  • Decrease by factor: Size reduces by fixed ratio. Example: Binary Search (n → n/2)
  • Variable decrease: Reduction varies. Example: Euclid's algorithm

Key difference from D&C: Only ONE subproblem, not multiple. No combine step.

2. Insertion Sort (4 marks)

Idea: To sort n elements, assume first n–1 are sorted, then insert the nth into correct position.

Pseudocode:
InsertionSort(A[0..n-1]):
  for i = 1 to n–1:
    key = A[i]; j = i – 1
    while j ≥ 0 and A[j] > key: A[j+1] = A[j]; j--
    A[j+1] = key

Trace on [5, 2, 8, 1, 9]:

  • i=1: key=2. 5>2 → shift. → [2, 5, 8, 1, 9]
  • i=2: key=8. 5<8, stop. → [2, 5, 8, 1, 9]
  • i=3: key=1. 8>1, 5>1, 2>1, stop. → [1, 2, 5, 8, 9]
  • i=4: key=9. 8<9, stop. → [1, 2, 5, 8, 9] ✓

Complexity: Best=O(n), Worst=O(n²), Average=O(n²). Space=O(1). Stable=Yes. Adaptive=Yes.

3. Topological Sorting (4 marks)

Definition: Linear ordering of vertices of a DAG such that for every edge u→v, u appears before v. Only exists for DAGs (no cycles).

Source Removal Algorithm:
TopologicalSort(G):
  result = []; compute in-degrees for all vertices
  while G has vertices:
    find vertex v with in-degree = 0
    append v to result
    remove v and its edges; update in-degrees
  return result

Example: Graph: A→C, B→C, B→D, C→E, D→F, E→F

  • In-degrees: A=0, B=0, C=2, D=1, E=1, F=2
  • Step 1: Remove A → in-degree C: 1. Order: [A]
  • Step 2: Remove B → in-degree C: 0, D: 0. Order: [A,B]
  • Step 3: Remove C → in-degree E: 0. Order: [A,B,C]
  • Step 4: Remove D → in-degree F: 1. Order: [A,B,C,D]
  • Step 5: Remove E → in-degree F: 0. Order: [A,B,C,D,E]
  • Step 6: Remove F. Final: [A,B,C,D,E,F] ✓

Complexity: O(V+E). Applications: Task scheduling, course prerequisites, compiler dependency resolution.

Q10 Compare and contrast all sorting algorithms studied: Bubble Sort, Selection Sort, Insertion Sort, Merge Sort, and Quick Sort. Discuss when to use each algorithm. [10 marks]
1. Brief Description of Each (2 marks)
  • Bubble Sort: Repeatedly swap adjacent elements; largest bubbles to end each pass
  • Selection Sort: Find minimum, swap to front; minimize writes
  • Insertion Sort: Build sorted array one element at a time by insertion
  • Merge Sort: Divide in half, sort recursively, merge; always O(n log n)
  • Quick Sort: Partition around pivot; fast in practice but O(n²) worst case
2. Complete Complexity Comparison Table (3 marks)
  • Bubble: Best=O(n)*, Avg=Θ(n²), Worst=Θ(n²), Space=O(1), Stable=Yes, In-place=Yes
  • Selection: Best=Θ(n²), Avg=Θ(n²), Worst=Θ(n²), Space=O(1), Stable=No, In-place=Yes
  • Insertion: Best=O(n), Avg=O(n²), Worst=O(n²), Space=O(1), Stable=Yes, In-place=Yes
  • Merge: Best=Θ(nlogn), Avg=Θ(nlogn), Worst=Θ(nlogn), Space=O(n), Stable=Yes, In-place=No
  • Quick: Best=O(nlogn), Avg=O(nlogn), Worst=O(n²), Space=O(logn), Stable=No, In-place=Yes

*Bubble Sort best case O(n) requires early-exit flag implementation

3. Key Differences (2 marks)
  • Stability: Bubble, Insertion, Merge = Stable. Selection, Quick = NOT stable
  • Space: Merge = O(n) extra. Others = O(1) (Quick needs O(log n) for call stack)
  • Predictability: Merge Sort always Θ(n log n). Quick Sort varies (O(n²) possible)
  • Adaptivity: Insertion, Bubble (with flag) = adaptive. Others = not adaptive
4. When to Use Each (2 marks)
  • Bubble Sort: Teaching/educational only. Almost never used in practice.
  • Selection Sort: When memory writes are expensive (flash storage); small n
  • Insertion Sort: Nearly sorted data; small arrays (n ≤ 20); online sorting; used in TimSort
  • Merge Sort: Need guaranteed O(n log n); stability required; linked lists; external sorting
  • Quick Sort: General-purpose in-place sorting on arrays; best average performance; used in most library sort functions
5. Conclusion (1 mark)

For general use: Quick Sort (fast in practice). When stability matters: Merge Sort. For small/nearly-sorted data: Insertion Sort. Modern languages (Python, Java) use TimSort = Merge Sort + Insertion Sort hybrid for best practical performance.

Q11 Explain the Divide and Conquer strategy. Describe Binary Search with pseudocode, trace, and recurrence analysis. Prove its time complexity using backward substitution. [10 marks]
1. Divide and Conquer Strategy (2 marks)

Divide and Conquer works in three phases:

  • Divide: Split problem of size n into a subproblems of size n/b
  • Conquer: Recursively solve each subproblem (base case: solve directly)
  • Combine: Merge solutions of subproblems to get overall solution

General recurrence: T(n) = aT(n/b) + f(n). Examples: Binary Search (a=1), Merge Sort (a=2), Strassen (a=7, b=2).

2. Binary Search — Definition (1 mark)

Efficiently search a sorted array by repeatedly comparing the target with the middle element and discarding the half that cannot contain the target. Each step eliminates half the remaining elements.

3. Pseudocode (2 marks)

BinarySearch(A[0..n-1], key): [Iterative]
  low = 0; high = n – 1
  while low ≤ high:
    mid = (low + high) / 2
    if A[mid] == key: return mid
    elif A[mid] < key: low = mid + 1
    else: high = mid – 1
  return –1

4. Trace Example (2 marks)

Array A = [2, 5, 8, 12, 16, 23, 38, 56, 72, 91], key = 23

  • low=0, high=9, mid=4: A[4]=16 < 23 → low=5
  • low=5, high=9, mid=7: A[7]=56 > 23 → high=6
  • low=5, high=6, mid=5: A[5]=23 == 23 → Return 5 ✓
5. Recurrence and Backward Substitution (2 marks)
  • Recurrence: T(n) = T(n/2) + 1, T(1) = 1
  • T(n) = T(n/2) + 1
  • = T(n/4) + 1 + 1 = T(n/4) + 2
  • = T(n/8) + 3
  • = T(n/2ᵏ) + k
  • When n/2ᵏ = 1 → k = log₂n
  • T(n) = T(1) + log₂n = 1 + log₂n ∈ Θ(log n) ✓
6. Properties (1 mark)
  • Best: O(1). Worst/Avg: O(log n). Space: O(1) iterative, O(log n) recursive
  • Prerequisite: Array must be sorted
